# 1) 数据准备（`prepare_drone_highway.py`）—— 与精度相关且能改动的参数

这些参数直接决定训练时看到的点密度、几何细节和上下文覆盖，优先考虑：

* `CFG['grid_size']`（体素下采样分辨率）

  * 影响：越小保留越多细节，利于精度；越大去噪、加速但损失细节。
  * 推荐（偏精度）：**0.01–0.02 m（1–2 cm）**。你原先 0.02 是合适的；若想更精细可用 0.01，但会显著增大点数/IO。

* `CFG['block_size']` 与 `CFG['stride']`（切块与步长 / 重叠）

  * 影响：较大的 block 能保留更多上下文（有利识别长形/跨块目标），但点数/显存更高；重叠（block_size - stride）决定边界处信息冗余，越大越能避免边界错分。
  * 推荐（偏精度）：`block_size = 60–100 m`（若道路环境很长可更大），`stride = block_size - overlap`，`overlap = 20–30%`，例如 `block_size=80, stride=60`（25% overlap）。更大的 overlap 有助于边界投票融合（见推理策略）。

* `CFG['min_points']`

  * 影响：丢弃太稀的块。对于追求精度，可以适当降低以保留更多块（但可能带来噪声）。
  * 推荐：`min_points = 2000–5000` 视数据稠密度而定。你当前 5000 是保守的。

* 别忘了保存**原始高密度下采样后的 block（debug/检视）**，并把输入 `np.save` 格式里的 `label`、`intensity`、`rgb` 校验好（已做过）。

* 额外建议：**多尺度预处理**。为每个块同时生成两个分辨率版本（比如 grid=0.02 和 grid=0.04），训练时可把 multi-scale features 拼接进网络或做 multi-branch 网络，通常能提高精度。

```py

CFG = {
    'input_path': BASE_DIR / 'datasets/drone_highway/original_data', # 输入 .las 文件夹
    'output_path': BASE_DIR / 'datasets/test',           # 'datasets/drone_highway',              # 输出根目录
    'grid_size': 0.001,         # 2cm 体素下采样
    'block_size': 5,        # 空间切块大小 (米)
    'stride': 4,            # 步长 (米)，意味着有 10m 的重叠，（40m → 10m 重叠）
    'min_points': 2000,        # 若切块后点数少于此值，丢弃 (防止空块)
    'val_ratio': 5,            # 每 5 块取 1 块作为验证集 (空间离散划分)
    'features_dim': 9          # [x,y,z, r,g,b, i, ret_n, n_ret]
}

```

# 2) 数据加载（`data.py`）—— 保证随机性、IO 与稳定性的参数

关键点：当你有百万 / 千万级点时，IO 与内存管理重要，即便不追求速度也要稳。

* `cfg.num_points`（每个 sample 的点数）

  * 影响：越大能保留更多上下文与细节 → 精度提高，但占显存。
  * 推荐（偏精度、4080Super 24GB VRAM）：**`num_points` 在 65536–131072 之间尝试**。

    * 如果显存 OK，试 131072（更高精度）；否则 65536 作为稳健起点。
    * 若显存受限，用 `gradient accumulation`（见训练策略）来模拟更大 batch。

* `DataLoader.batch_size` 与 `num_workers`

  * 影响：batch_size 与 num_points 决定显存占用；num_workers 决定 CPU IO 并行。
  * 推荐：`batch_size` 初始设为 **1–4**（当 `num_points >= 65536` 时），先用 `1` 检查，然后增加到 `2/4` 看显存。`num_workers=8` 对你 64GB 内存 OK；若 IO 成为瓶颈，可设更高。

* `use_mmap`（上次建议）与缓存

  * 建议在 `CloudDataset` 中启用 `np.load(..., mmap_mode='r')` 或实现文件缓存（你有 64GB，可缓存若干块）。mmap 在文件很大时避免大量内存占用。

* 文件选择策略

  * 对精度有利的做法：**随机选择文件 + 随机采样点**（你已实现），但建议把 `file_choice` 改为 `random.choice(self.files)`（而非 `idx % len(files)`），保证更均一采样。
  * 可以实现**类别/困难样本重采样**（active learning 风格），让网络更多见稀有类别 / 边界样本。

* `collate_fn` 保持 `torch.stack` 即可。

# 3) 模型结构超参（`model.py`）—— 能直接提升精度的调整

你现在的 RandLA-Net 基本实现没大问题，但若你只关心精度，可以在这些点微调：

* `num_neighbors`（邻居数 K）

  * 影响：更大 K 捕获更大感受野（更多上下文），但可能模糊局部细节并增算力/内存。
  * 推荐（偏精度）：**16–32**；若你的场景需要广域上下文，试 `K=32`。

* `decimation`（d，编码器下采样倍率）

  * 影响：decimation 越小（如 2），在编码阶段保持更多点 → 提高保真与精度；越大（如4）加速但可能丢信息。
  * 推荐（偏精度）：**`decimation = 2`**（比原论文常用的 4 更保真，但消耗更大算力与内存）。

* 各层通道数（`LocalFeatureAggregation` 的输入/输出 dims）

  * 影响：增加通道数提升模型容量，有利精度，但易过拟合且显存大。
  * 推荐：在能承受显存的情况下，把 encoder/decoder 宽度提高 1.5–2×（例如把 8→16，16→32，64→128 等），但优先调 `decimation` 与 `num_points` 再考虑通道扩容。

* `SharedMLP` 中的 batchnorm/activation：保留即可，BN 的 momentum/eps 可微调但一般默认 OK。

* **注意：你的 model 里 `coords` 经常被 `.cpu()`** —— 这会大幅慢，但若你暂时不关心速度，可先无视；不过如果 `.cpu()` 拖慢到不可接受，会影响迭代速度（但你说不求速度）。仍建议尽快切换到 GPU knn 以保持训练可行（尤其当 num_points 很大）。

# 4) 训练超参与策略（`mytrain.py`）—— 直接影响收敛和精度

* `cfg.num_points` + `batch_size`（见上）

* `optimizer`：Adam 是稳健的选择；若进一步追求精度，可尝试 **AdamW**（带权重衰减）并适度加 `weight_decay = 1e-4`。

* `learning rate`（`--adam_lr`）

  * 推荐初始：**0.01** 对 Adam 有点高（但你之前用过）。更保守且更稳定的话使用 **0.001–0.005**。若你用 `batch_size` 很小，建议 lr 较小（0.001）。
  * 使用学习率预热（warmup）和余弦退火或 `ExponentialLR` 都可以。建议：`warmup`（100–500 steps） + `CosineAnnealingLR`。

* `scheduler_gamma`：你原来 0.95 每 epoch 衰减很平缓；适用但可尝试 `CosineAnnealing` 与更长训练（200–400 epochs）以提升精度。

* `epochs`：

  * 精度优先：**200–400 epochs**（如果训练时间不是问题）。

* `loss`：

  * `CrossEntropyLoss(weight=class_weights)` 是基本做法。若极度类不平衡，考虑 **Focal Loss**（γ=2）或 label-specific weights。使用 class-wise `alpha` 在 FocalLoss 上对少数类放大很有效。

* `compute_class_weights`：

  * 保持自动计算；也可人工放大稀有类权重（经验值 2–10×）。

* `gradient accumulation`：

  * 当 `num_points` 很大且 batch_size 需要保持小（1），可以使用梯度累积（`accumulation_steps`）来模拟大 batch，从而稳定优化与使用 larger lr schedule。

* `precision`：

  * 你偏好精度 → 使用 **FP32**（默认）。混合精度（AMP）常提高速度且有时微小影响精度，但一般安全。若你做极端数值训练可尝试 FP32 固定。

* `augmentation`（强烈建议）

  * 对精度影响大：随机旋转（Z 轴）、随机翻转、随机缩放（0.9–1.1）、随机抖动（小方差）、颜色通道扰动、点丢弃（dropout）等。对泛化和 mIoU 提升显著。
  * 对道路类场景，注意保持物体尺度（缩放幅度不能过大）。

* `checkpointing / validation`：

  * 多保存 checkpoint 并在验证集上监控 `mIoU`。采用早停或保存最佳 mIoU 的 checkpoint（而不是简单按 epoch 间隔保存）。

# 5) 推理 / 验证策略（可提高最终精度的技巧）

* **滑窗重叠投票**：训练时使用 overlapping blocks；推理时对覆盖同一点的多个块进行概率平均或多数投票，大大提升边界精度。（配合更大 overlap）
* **Test-time augmentation（TTA）**：多次旋转/翻转输入并平均预测。
* **模型集成**：训练多个不同 seed/超参模型并 ensemble（平均 logits）通常能提高 mIoU。
* **CRF / 后处理**：简单的空间连通或条件随机场后处理可改善噪声标签（可选）。

# 7) 其他工程建议（收敛与稳定性）

* 先用 **小样本调试 pipeline（1–2 batches）**，确保形状/dtype正确，再上全训练。
* 打开 TensorBoard 监控 loss / mIoU / per-class IoU，观察是否过拟合、学习率是否合适。
* 若训练不稳定，先降低 lr（0.0005），增加 weight decay（1e-4），或使用梯度裁剪 (`torch.nn.utils.clip_grad_norm_`).
* 如果 IO 成瓶颈，考虑把部分预处理后的 `.npy` 缓存到内存（你有 64GB）或使用 faster SSD。

# 8) 小结（关键优先级指南）

优先级（对精度影响最大 → 次之）：

1. `num_points`（每个 sample 点数）↑
2. `decimation` ↓（更小的下采样因子）
3. `neighbors` ↑（捕获更大上下文）
4. `block_size` ↑ + `overlap` ↑（上下文与重叠投票）
5. 数据增强与 longer training（epochs）
6. loss 增强（Focal / class weights）
7. 通道宽度 ↑（模型容量）
